# 03 — Bilingual i18n: add Chinese (zh) and prove the two-locale path

- **Status:** DONE
- **Owner:** Codex
- **Blocked by:** — (task 01 merged)
- **Allowed new deps:** none (gen_l10n + flutter_localizations already wired in task 01)

## Goal
Bilingual zh/en is a core product requirement, but the scaffold only ships `app_en.arb`. Add a
matching Chinese translation file and make sure the generated two-locale localization actually
works (device locale → correct strings, with sensible fallback). No new UI — this is purely
making the existing strings bilingual and proving the plumbing.

## Scope
- in:
  - Add `lib/l10n/app_zh.arb` with a Chinese translation for **every** key currently in
    `app_en.arb` (keep the same keys; ARB only needs `@`-descriptions in the template `app_en.arb`).
  - Run `flutter gen-l10n` (or rely on `flutter: generate: true`) so `app_localizations_zh.dart`
    is produced; ensure `AppLocalizations.supportedLocales` includes `zh`.
  - Confirm `MaterialApp.router` already exposes `supportedLocales` from generated code (it does
    after task 01) so the new locale is picked up automatically.
  - Add a widget test that pumps the app under `Locale('zh')` and asserts the Chinese tab labels
    render; keep the existing English test working (English remains the fallback).
- out:
  - No language-switcher UI / settings screen (that belongs to a later settings task).
  - No new user-facing strings beyond the keys that already exist.
  - Do not touch routing, theme, or any feature logic.

## Acceptance criteria
- [ ] `lib/l10n/app_zh.arb` exists and translates every key in `app_en.arb` (no missing keys).
- [ ] Generated `app_localizations_zh.dart` builds; `supportedLocales` contains both `en` and `zh`.
- [ ] A widget test under `Locale('zh')` asserts Chinese labels; the English test still passes.
- [ ] Device set to an unsupported locale falls back to English without crashing.
- [ ] `dart format` / `flutter analyze` / `flutter test` all clean.
